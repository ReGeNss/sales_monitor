# Sales Monitor

Система моніторингу цін на продукти в українських маркетплейсах (АТБ, Fora, Silpo).

---

## Запуск проекту

### Вимоги

- Docker та Docker Compose
- Go 1.25+
- Node.js 20+ та npm
- Playwright

### Змінні середовища

Створіть `.env` файл у корені проекту на основі прикладу:

```env
DB_HOST=mariadb
DB_PORT=3306
DB_ROOT_PASSWORD=root_password
DATABASE_NAME=sales_monitor
DB_USER_NAME=app_user
DB_USER_PASSWORD=app_password

REDIS_HOST=redis_queue
NOTIFICATION_QUEUE_KEY=notification_queue

JWT_SECRET=your_jwt_secret

API_PORT=3000
API_CORS_ORIGIN=http://localhost:3000
API_PREFIX=api

PUSHGATEWAY_PORT=9091
PROMETHEUS_PORT=9090
SCRAPER_LOGS_PORT=8081
SCRAPER_LOGS_PUBLIC_URL=http://localhost:8081
```

### Запуск через Docker Compose

```bash
docker compose up -d

# Зупинити
docker compose down
```

Сервіси після запуску:
| Сервіс | URL |
|---|---|
| REST API | `http://localhost:3000/api` |
| phpMyAdmin | `http://localhost:8080` |
| Prometheus | `http://localhost:9090` |
| Pushgateway | `http://localhost:9091` |
| Scraper Logs | `http://localhost:8081` |

### Локальний запуск API (без Docker)

```bash
# Встановити залежності
npm install

# Зібрати пакети
npm run build --workspace=@sales-monitor/database
npm run build --workspace=@sales-monitor/common

# Запустити API у watch-режимі
npm run dev:api
```

### Локальний запуск Go-сервісів

```bash
# Scheduler
go run ./scheduler/...

# Notifications Service
go run ./notifications_service/...
```

## Автентифікація та акаунт

### UC-01 — Реєстрація нового користувача

**Актор:** Відвідувач

**Передумови:**
- Користувач не має акаунту
- Сервіс доступний

**Основний сценарій:**
1. Користувач надсилає `POST /auth/register` з логіном та паролем
2. Система перевіряє, що логін унікальний
3. Система хешує пароль і зберігає нового користувача в БД
4. Система повертає JWT-токен

**Альтернативні сценарії / помилки:**
- Логін вже зайнятий → `409 Conflict`
- Пароль не відповідає мінімальним вимогам → `400 Bad Request`
- Відсутній логін або пароль → `400 Bad Request`

---

### UC-02 — Вхід в систему

**Актор:** Зареєстрований користувач

**Передумови:**
- Користувач має акаунт

**Основний сценарій:**
1. Користувач надсилає `POST /auth/login` з логіном та паролем
2. Система знаходить користувача за логіном
3. Система перевіряє хеш пароля
4. Система повертає JWT-токен

**Альтернативні сценарії / помилки:**
- Логін не знайдено → `401 Unauthorized`
- Невірний пароль → `401 Unauthorized`
- Відсутні поля → `400 Bad Request`

---

### UC-03 — Гостьовий доступ

**Актор:** Відвідувач

**Передумови:**
- Немає

**Основний сценарій:**
1. Користувач надсилає `POST /auth/guest`
2. Система генерує гостьовий JWT-токен без прив'язки до акаунту
3. Користувач отримує доступ до перегляду продуктів і цін (без улюблених)

**Альтернативні сценарії / помилки:**
- Спроба викликати захищений ендпоінт (улюблені) → `403 Forbidden`

---

### UC-04 — Підключення push-сповіщень

**Актор:** Зареєстрований користувач

**Передумови:**
- Користувач авторизований (JWT)
- На пристрої дозволено push-сповіщення

**Основний сценарій:**
1. Клієнтський додаток отримує Firebase token (NFToken)
2. Додаток надсилає token на сервер
3. Система зберігає NFToken у профілі користувача
4. Надалі сповіщення надсилаються на цей пристрій

**Альтернативні сценарії / помилки:**
- Невалідний або порожній token → `400 Bad Request`
- Token вже зареєстровано → оновлення без помилки

**Передумови:**
- Є хоча б один продукт у базі

**Основний сценарій:**
1. Користувач надсилає `GET /products` з опціональними параметрами: `category`, `brand`, `search`, `page`, `limit`

## Сповіщення про зниження ціни

### UC-22 — Отримати push-сповіщення при зниженні ціни

**Актор:** Зареєстрований користувач

**Передумови:**
- Користувач має NFToken (Firebase) збережений у профілі
- Продукт або бренд є в улюблених користувача

**Основний сценарій:**
1. Система (Notifications Service) виявляє зниження спеціальної ціни
2. Знаходить користувачів, у яких цей продукт або бренд в улюблених
3. Надсилає Firebase push-сповіщення на збережений NFToken
4. Користувач отримує сповіщення на пристрій

**Альтернативні сценарії / помилки:**
- NFToken відсутній або застарів → сповіщення не доставлено, помилка логується
- Продукт не в улюблених → сповіщення не надсилається

---

### UC-23 — Поставити продукт в чергу сповіщень

**Актор:** Система (Notifications Service)

**Передумови:**
- Нова ціна збережена в БД
- Redis доступний

**Основний сценарій:**
1. Scraper зберігає нову ціну
2. Система порівнює нову спеціальну ціну з попередньою
3. Якщо нова ціна нижча — формується NotificationTask (brandId, brandName, products)
4. Task надсилається до Redis-черги через LPUSH

**Альтернативні сценарії / помилки:**
- Попередньої ціни немає (перший запис) → сповіщення не надсилається
- Ціна не змінилась або зросла → сповіщення не надсилається
- Redis недоступний → помилка логується, task втрачається

---

### UC-24 — Відправити push-сповіщення через Firebase

**Актор:** Система (Notifications Service)

**Передумови:**
- У Redis-черзі є NotificationTask
- Firebase налаштований і доступний

**Основний сценарій:**
1. Notifications Service читає task з черги через Redis BLPOP
2. Знаходить у БД користувачів з відповідними улюбленими та NFToken
3. Формує Firebase Cloud Messaging повідомлення
4. Надсилає push на кожен NFToken

**Альтернативні сценарії / помилки:**
- Firebase повертає помилку для конкретного token → логується, решта надсилаються
- Черга порожня → сервіс чекає наступного task (blocking read)

## Збір даних

### UC-25 — Запустити скрапінг маркетплейсу за розкладом

**Актор:** Система (Scheduler)

**Передумови:**
- Конфігураційний YAML-файл існує і валідний
- Scraper-бінарник доступний

**Основний сценарій:**
1. Scheduler завантажує конфігурацію з YAML
2. За cron-розкладом запускає scraper як subprocess з потрібними параметрами
3. Логує результат виконання

**Альтернативні сценарії / помилки:**
- YAML-конфігурація невалідна → Scheduler логує помилку, не запускає jobs
- Subprocess завершився з помилкою → помилка фіксується в логах

---

### UC-26 — Зібрати ціни та продукти з АТБ

**Актор:** Система (Scraper)

**Передумови:**
- Playwright доступний
- Сайт АТБ доступний

**Основний сценарій:**
1. Scraper відкриває браузер через Playwright
2. Переходить по сторінках каталогу АТБ
3. Витягує: назву, регулярну та спеціальну ціну, зображення, об'єм/вагу
4. Передає продукти до сервісу обробки

**Альтернативні сценарії / помилки:**
- Сайт недоступний або змінив структуру → scraper завершується з помилкою, логується

### UC-29 — Дедублікувати продукти між маркетплейсами

**Актор:** Система (Scraper)

**Передумови:**
- Зібрані «сирі» продукти передані до ProductService

**Основний сценарій:**
1. Система нормалізує назву продукту (NameFingerprint)
2. Шукає існуючий продукт за fingerprint у кеші Redis та БД
3. Якщо знайдено — пов'язує MarketplaceProduct з існуючим Product
4. Якщо не знайдено — застосовує fuzzy matching для пошуку схожих

- Fuzzy matching дає хибнопозитивний результат для схожих, але різних продуктів → конфігурація `words_to_differentiate` виключає їх
- Помилка БД → операція логується, продукт пропускається
